# 2026-06-24

Built the v1 review spine from the future spec/work order. Durable points:

- `canon_path` is required in config; Terminus does not default to a sibling
  canon path, so operators must choose the vault explicitly.
- Review records intentionally live under user-level `log_destination`, outside
  the subject repo, because `_prompt.md` embeds private quality bodies.
- The selected quality set is frozen at dispatch and written into `status.json`
  and `_findings.md`; collection classifies from that start-time set, not from a
  reloaded rubric.
- `theharnessbody/record` is now the durable-record writer Terminus depends on.
  It remains domain-neutral; Terminus-specific audit material goes through
  generic markdown `Section`s.

## Clean-tree review promotion

- A bare `terminus review` on a clean tree built a 0-file working-tree
  changeset, which narrowed away every territory-scoped quality and returned a
  vacuous `clean`. The CLI now promotes a *defaulted* working-tree review to
  `full` when the tree is clean (`cmd/terminus/review.go`).
- This lives in the CLI on purpose, not the broker. The broker/MCP `start_review`
  path is left untouched so an agent that asks for `working-tree` gets exactly
  that — silently promoting an agent's request to a full-repo review is a worse
  surprise than an empty one. Don't "centralize" this into the broker.
- Gated on `cmd.Flags().Changed("kind")`: an explicit `--kind working-tree` is
  honored even on a clean tree. The clean check reuses `changeset.WorkingTree`
  rather than a separate git probe so the CLI's "clean" can't drift from the
  broker's dirty-set definition.
- Still live: the deeper wart is that an empty changeset reports `clean` at all.
  The explicit-working-tree and MCP paths still return a vacuous `clean` on an
  empty selection; giving "nothing reviewed" a distinct non-`clean` signal is the
  unbuilt complementary fix.

## Reviewer prompt + canon decoupling

- Dropped the quality `convention` frontmatter field (was in `canon.Head`). It
  was provenance-only, never loaded, and pointed back into the grimoire — the one
  quality that set it pointed at the *wrong* note. The canon is the self-contained
  sole authority; bodies must not depend on anything outside the canon. The field
  removal is load-bearing-coupled to the canon repo: `canon.Head` uses strict
  unknown-key rejection (`+extra`), so any `convention:` line left in a canon
  quality file now fails parsing. The terminus-canon `df-logging.md` line was
  removed in the same change.
- The prompt now points the reviewer at each quality body's `discrimination` and
  `boundary` sections as binding flag/skip calibration. This couples the prompt to
  the canon body template — hedged two ways so it degrades gracefully: the prompt
  says "or equivalent prose" and falls back to `statement`/`why` when those
  sections are absent, and the template is now documented in the terminus-canon
  README so it's a contract, not an assumption. Don't tighten the prompt to assume
  the sections always exist.
- Deliberate non-changes (don't "add" these — they're intentional v1 design):
  no reviewer-chosen severity, no verdict/advisory_notes arrays, no numeric
  finding budget. Blocking-ness stays a property of the quality, computed
  server-side; the "prefer fewer, material" line carries the anti-quota intent
  without a cap.

## Changeset as starting points (not a fence)

- Removed `findings.CheckScope` and its broker call. Findings are no longer fenced
  to the changeset; a finding may name any file. The reviewer is prompted to treat
  changed files as starting points and follow a change outward (caller, dependent)
  and file where the problem lives. This was a deliberate relaxation, not an
  oversight — do not "restore" scope enforcement.
- `normalizeFile` and the `path/filepath` import went with `CheckScope` (only
  consumer). The surviving validation hard-fails are: schema, duplicate id,
  `CheckAttribution` (finding must cite a selected quality id). Note the asymmetry
  that follows: the reviewer can range into a file whose territory no selected
  quality covers, but can only *report* there under a quality the changeset already
  pulled in — attribution still gates findings to selected qualities.
- Diff and deletions are KEPT (reversing the direction of the prior journal note
  about dropping them). Rationale: in the entry-points model they are the seed for
  tracing a change's blast radius, not the boundary of review. A deleted file is a
  legitimate trace seed (what was removed → who depended on it). Don't drop them.
- Open and explicitly unsettled: whether the targeted-vs-full distinction should
  exist at all, vs. every review being effectively full with changed files as a
  spotlight. Michael's instinct leans full; left to real-world experience on a
  richer canon (only two broad-territory qualities today, so the distinction barely
  bites). Captured in docs/future/terminus-follow-ons.md — do not treat this change
  as settling that question.
- Deliberate non-action: a live review flagged that `changeset_kind`/`ChangesetKind`
  (MCP input, broker, `changeset` package, status field) still encodes the old
  fenced model now that selections are "starting points." We considered and
  DEFERRED the rename — doing it would hard-commit the public API to a model we
  just marked unsettled. The reviewer will keep re-raising this (terminus has no
  disposition capture yet); don't "fix" it until the targeted-vs-full question
  settles toward starting-points.

## Named rubrics per project

- `LoadRubric`/`LoadProjectRubric` now take a rubric name; path is
  `projects/<project>/<name>.yaml`, default `canon.DefaultRubric` ("rubric").
  Empty request → default; threaded through CLI `--rubric`, MCP `start_review`
  `rubric`, broker job, and recorded in status.json/result.json + the `_findings.md`
  "Selected qualities" section.
- Each named rubric still declares `project.repo` and is validated against the
  repo basename (a named rubric with the wrong repo fails like the default does).
  Two guards on the name, intentionally redundant: `rubricFileName` rejects
  `/`, `\`, `..`; `ensureContained` is defense-in-depth on the resolved path.
- Deliberate scope choices (don't "complete" these without a real need):
  flat layout (no `rubrics/` subdir), independent hand-authored lists (no
  include/compose), and CLI-only discovery via `terminus rubrics` — there is no
  MCP `list_rubrics` tool, agents pass a known name. All three are provisional and
  written up in docs/future/terminus-follow-ons.md.
- Blocking is per-rubric by construction (the `blocking` flag lives on each rubric
  entry), so the same quality can block under one rubric and be advisory under
  another — intended, not an accident to normalize away.

## mcpserver composition moved to wiring (dogfooded fix)

- `mcpserver.New` now takes a `*broker.Broker` (was `*config.Config`) and only
  wraps it as an MCP server; broker construction moved to `wiring.NewBroker`,
  called by the `cmd` layer (both serve and review now share it). `mcpserver` no
  longer imports `internal/config` or `internal/wiring` — the adapter stopped
  depending on the composition root.
- Found by terminus reviewing *itself*: a full review under terminus's own rubric,
  after adopting `clean-layering`, flagged the leak as a blocking finding. Good
  closed loop — keep terminus's own review clean under its rubric.
- Dropping the old `cfg.Validate()` + nil-check from `mcpserver.New` is safe, not
  an oversight: `config.Load` already validates (config.go) and never returns a
  nil cfg without an error, so both the serve and review paths were already
  validated at load time. The mcpserver checks were redundant.
- Side benefit, not yet taken: `mcpserver.New` can now be tested with a stub
  broker (previously it built its own). No test added — flagging it as available.
