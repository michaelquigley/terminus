# 2026-06-24

Built the v1 review spine from the future spec/work order. Durable points:

- `canon_path` is required in config; Terminus does not default to a sibling
  canon path, so operators must choose the vault explicitly.
- Review records intentionally live under user-level `log_destination`, outside
  the subject repo, because `_prompt.md` embeds private quality bodies.
- The selected quality set is frozen at dispatch and written into `status.json`
  and `_findings.md`; collection classifies from that start-time set, not from a
  reloaded rubric.
- `theharnessbody/record` is now the durable-record writer Terminus depends on.
  It remains domain-neutral; Terminus-specific audit material goes through
  generic markdown `Section`s.

## Clean-tree review promotion

- A bare `terminus review` on a clean tree built a 0-file working-tree
  changeset, which narrowed away every territory-scoped quality and returned a
  vacuous `clean`. The CLI now promotes a *defaulted* working-tree review to
  `full` when the tree is clean (`cmd/terminus/review.go`).
- This lives in the CLI on purpose, not the broker. The broker/MCP `start_review`
  path is left untouched so an agent that asks for `working-tree` gets exactly
  that — silently promoting an agent's request to a full-repo review is a worse
  surprise than an empty one. Don't "centralize" this into the broker.
- Gated on `cmd.Flags().Changed("kind")`: an explicit `--kind working-tree` is
  honored even on a clean tree. The clean check reuses `changeset.WorkingTree`
  rather than a separate git probe so the CLI's "clean" can't drift from the
  broker's dirty-set definition.
- Still live: the deeper wart is that an empty changeset reports `clean` at all.
  The explicit-working-tree and MCP paths still return a vacuous `clean` on an
  empty selection; giving "nothing reviewed" a distinct non-`clean` signal is the
  unbuilt complementary fix.
